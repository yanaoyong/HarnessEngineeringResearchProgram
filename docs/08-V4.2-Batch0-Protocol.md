## 5. Experiment Identity and Run Identity

V4.2 新实验使用：

```text
EXP-C01-01
EXP-C08-02
EXP-C18-01
```

一个 Experiment 可以包含多个 Run。Experiment ID 表示研究设计，Run metadata 表示一次可复查执行。Run 之间不得仅靠文件名隐含 Host、Provider、Model 或配置差异。

### Batch 0 审查后澄清：Result 层级

本澄清修复模板表达歧义，不改变 Cycle、Batch、术语、Evidence taxonomy 或 Support Level 冻结决策：

- `SUPPORT / REJECT / INCONCLUSIVE` 是 Hypothesis 的 Experiment Result，只能在 Experiment Record 中按预注册规则裁决。
- 单次 Run Metadata 只记录 execution status、acceptance status、observation、限制与是否纳入 Experiment Result；不得把一次 Run 直接写成 Hypothesis Result。
- Experiment 必须声明 `HYPOTHESIS_RESULT` 或 `OBSERVATION_ONLY`。后者允许不填写 Hypothesis，以 `COMPLETE / PARTIAL / INVALIDATED` 记录 Observation Outcome，并将 Experiment Result 写为 `NOT APPLICABLE · OBSERVATION ONLY`；不得用 `INCONCLUSIVE` 代替“不裁决 Hypothesis”。
- 一个 Experiment 若包含多个不可互相替代的 Host、task instance 或其他 stratum Result，必须在执行前声明 Result unit、stratum key 与 aggregation rule。
- 不存在合理聚合规则时，只报告 stratum Result，并将 Experiment-level Result 标为不适用；不得为了模板完整而制造统一结论。

Run 的 `repository.commit` 绑定 task fixture。若实验还加载独立 Harness baseline / variant，Run 必须另行绑定 cell ID、Harness revision、repository / distribution revision 与 implementation artifact IDs；Rule / Skill / Check / Adapter revision 不能替代完整 Harness-under-test 身份。

现有 `EXP-Wxx-yy` 保持 V4.1 historical ID，直到对应内容 Batch 迁移。已经产生的历史 Evidence 不得为了编号整齐而重新编号；迁移时通过 legacy mapping 关联新 Cycle。

## 6. Research Workspace Naming

新工作区按 Cycle 命名：

```text
research/cycles/cycle-01/
├── research-note.md
├── experiments/
└── evidence/
```

内容 Batch 可以在实验执行前，为本 Batch 覆盖的 Cycle 准备工作区目录。目录准备不表示实验已经执行、Evidence Claim 已经建立或 Exit Criteria 已经满足。

Batch 0 没有创建任何 Cycle 目录。Batch 1 只准备 `cycle-01` 与 `cycle-02`；不得提前创建尚未进入对应内容 Batch 的 Cycle 3–18 目录。

## 9. Stable Task Suite Boundary

T01–T03 是稳定的任务语义，不是固定公开 benchmark 数据集：

- `T01 · Engineering Constraint`
