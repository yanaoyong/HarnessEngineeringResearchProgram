## Agent System Terms

### Agent

以 Model 为推理核心，围绕目标维护交互状态、选择 Action、调用 Tool、接收 Observation 并判断继续或停止的执行实体。Agent 不等于 Model，也不自动包含完整 Harness。

### Coding Agent Host

承载 Coding Agent 的产品或运行环境，提供 Agent loop、context、tool execution、permission、session、extension surface 与用户交互。Claude Code、Codex、Qwen Code、OpenCode 在当前计划中属于 Host。

### Harness

围绕 Agent 与 Host 组织工程行为的约束和能力集合，包括可移植语义、Host Adapter、Rules、Skills、Hooks、Checks、workflow、evidence 与 project-specific governance。Harness 不等于 Host，也不等于单个 Plugin。

### Model

执行推理和生成的具体模型标识。Model behavior 必须与 Host behavior、Provider behavior 分开观察。

### Provider

提供模型访问、鉴权、路由、限额与 endpoint 的服务方。相同 Model 经不同 Provider 或 endpoint 可能表现不同。

### Configuration

影响一次执行的显式设置集合，包括 Host、Provider、Model、tool、permission、sandbox、instruction、sampling、endpoint 与 feature flags。Configuration 必须可快照或可追溯。

### Tool

Agent 可以请求调用的外部能力及其输入输出契约，例如读文件、执行命令、搜索或访问服务。Tool availability 与 Tool behavior 由 Host、Adapter、Configuration 和环境共同决定。

## Guidance and Extension Terms

### Instruction

提供给 Model 的自然语言方向或约束。Instruction 依赖模型理解与遵循，本身不等于确定性执行或阻断。

### Rule

在明确作用域内描述必须、禁止或优先行为的持久工程约束。Rule 可以通过 Instruction 表达，也可以由 Check 或 Hook 强制；定义 Rule 时必须说明其 enforcement mode。

### Skill

按需发现和加载的程序性知识或工作流程，用于改变 Agent 处理某类任务的方式。Skill 被发现或加载不等于被正确执行，也不等于结果有效。

### Hook

由 Host 生命周期事件或确定性触发点调用的逻辑。Hook 可以观察、注入、验证或阻断；具体事件与能力属于 Host Contract，不假定跨 Host 等价。

### Plugin

某个 Host 可安装、分发或加载的扩展包。Plugin 可以携带多种 Harness Artifact，但 Plugin 的包装格式和加载机制通常是 Host-specific。

## Portability Terms

### Portable Semantic Contract

独立于具体 Host 文件形状的行为语义，包括 intent、trigger、required context、procedure、evidence、completion、failure route 与 limitations。Portable 表示语义可映射，不保证跨 Host 行为完全一致。

### Host Adapter

把 Portable Semantic Contract 映射到特定 Host 的 instruction、tool、hook、plugin、permission 或 lifecycle surface 的适配层。Adapter 必须显式记录无法映射和降级行为。

### Provider Profile

对一次可复用 Provider 条件的版本化描述，包括 Provider、endpoint type、鉴权类别、路由、Model ID、限制和已知差异。它不包含秘密值。

### Provider-dependent Behavior

由 Provider、endpoint、路由、限额或协议转换导致，不能仅归因于 Host 或 Model 的行为差异。

### Host-specific Capability

依赖某一 Host 独有或无可接受语义映射的能力。它可以被 Harness 使用，但不得被伪装为 Portable Semantic Contract。

### Unsupported / Unknown

`Unsupported` 表示在绑定版本和 profile 下，有足够证据表明目标能力不存在或不能满足契约；`Unknown` 表示证据不足。未知不是不支持，不支持也不是永久结论。

## Evidence Terms

### Contract Evidence

来自官方产品文档、正式规范、官方 release information 或其他权威 Contract surface 的证据，用于说明公开承诺的能力与边界。

### Source Evidence

来自已验证权威归属、版本和范围的源码证据，用于说明特定 revision 的实现。Source Evidence 不自动证明部署行为、未来 Contract 或所有发行版本。

### Behavior Evidence

由绑定 Host、Provider、Model、Configuration 和时间的直接运行、Session、Trace 或 Tool observation 产生的证据，用于说明该条件下实际发生了什么。

### Project Evidence

来自目标项目真实 Change、failure、review、test、CI、artifact drift 或维护记录的证据，用于说明问题是否真实存在以及设计是否与项目相关。

### Enterprise Fact

由可验证部署、运维、安全、采购、审计或组织约束支持的特定企业事实。Enterprise Fact 只适用于记录的组织和环境，不自动构成官方 Contract、架构结论或普遍法律合规。

### Community Claim

来自社区文章、逆向工程、非官方仓库、客户端指纹、第三方 patch 或未经权威验证陈述的主张。它只能作为调查线索或 Open Question，不能证明官方 Contract 或 Architecture。

## Research Terms

### Mental Model

研究者对系统机制的当前解释。Mental Model 必须标记版本并接受证据修订；它不是证据本身。

### Hypothesis

基于 Mental Model 和 Evidence 提出的可证伪判断，必须包含适用边界以及可观察的支持、反驳或不确定信号。

### Exit Criteria

结束当前 Cycle、实验或研究分支所需满足的可观察条件。Exit Criteria 控制研究停止点，不等于成功标准。

### Route Review

每 2–4 个 Cycle 对问题、Mental Model、Hypothesis、Evidence、项目池和路线有效性进行的复盘。它可以调整路线，但必须记录原因。

### Support Level

对绑定 capability scope 与运行条件的证据成熟度分级 S0–S4。它不是 Host 的永久总评分，也不是产品排名、生产就绪或法律合规声明。
